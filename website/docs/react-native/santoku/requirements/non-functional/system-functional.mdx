モバイルアプリケーションのシステム方式に関する事項としては以下とします。

## 要件内容

- フレームワーク選定について
- アプリ方式について
- クラウドサービス・ソフトウェア製品選定について
- このアプリで利用するクラウドサービス・ソフトウェア製品について
    - Firebaseサービス

### フレームワーク選定について

フレームワークを追加する場合の選定基準は[デジタル・ガバメント推進標準ガイドライン][gideline]に照らし合わせ以下とします。

  - 当該フレームワークの保守及び技術サポートの調達が困難でないこと
  - 当該フレームワークのライセンス条項によって採用後に訴訟リスクなどが生じないこと

今回のアプリでは開発フレームワークとしてRect Nativeを採用しています。
これは先に記述した選定基準を以下のように満たしています。

- JavaScriptライブラリによるモバイルアプリケーション開発が可能であり、SI事業者に多く在籍している既存Webエンジニアへの学習コストが低い。
- 開発元であるFacebook内部でも使用されておりGitHub上でのスター数も上位に位置している。
- MITライセンスが適用されており採用後の法的トラブルに対する必要コストが低い。

### アプリ方式について

アプリ方式については以下のように定義されます。

- ネイティブアプリ

    ここでの「ネイティブアプリ」は、iOSであればObjectiveーC/Swift、AndroidであればKotlin/Javaで開発というようにプラットフォーム専用に開発したアプリを指します。
    上記の通りプラットフォームにより開発言語が異なるのでiOS/Android両方に対応したアプリを開発する場合それぞれで開発が必要になるため、単純計算で2倍の開発工数がかかる想定となりますが専門エンジニアの在籍状況によってはこちらの方が生産性が高いとされます。

- クロスプラットフォームアプリ

    特定の開発フレームワークを用いて1つのソースコードでiOS/Android両方で動作するアプリをここでは「クロスプラットフォームアプリ」と称します。  
    React NativeのようにWeb技術をベースにしたフレームワークもあるため専門エンジニアが少なくても生産性が確保できたり、フレームワークによってベースの開発はiOS/Androidで一本化されるためそれぞれのプラットフォーム向けに開発するよりも開発工数の低減が期待できる反面、OSバージョンアップによってフレームワーク側に問題が発生した場合フレームワークの対応を待っての修正となるため問題の即時対応が難しい場合があります。またネイティブ機能に関連した開発とリリースビルドはそれぞれの環境で行います。

このアプリではSI事業者に多く在籍しているWebエンジニアでも開発できることと、iOS/Android両方で開発することを踏まえて、生産性の確保と必要工数の削減を目指すことを目的に開発フレームワークに「クロスプラットフォームアプリ（React Native）」を採用します。

### クラウドサービス・ソフトウェア製品選定について

クラウドサービス・ソフトウェア製品選定においては、アプリケーションプログラムの動作、性能等に支障を来たさない範囲において、可能な限りOSS製品を活用します。
ただし、それらのOSS製品のサポートが確実に継続されていることを確認しなければならないものとします。

### このアプリで利用するクラウドサービスについて

#### Firebaseサービス

このアプリではクラッシュログの収集や利用状況のモニタリングのために[Google Firebase][firebase]サービスの以下サービスと連携を行います。

- [Google Analytics for Firebase][Analytics]
