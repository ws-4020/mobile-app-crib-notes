---
title: 画面の初期表示結果を確認する観点
---

ここでは単体テストの画面表示の初期表示結果を確認するテスト観点をまとめています。
それぞれのテスト観点を検証する際のテスト方法の種類は、以下のとおりです。

- （A）Jestを利用して、作成したテストコードで自動テストを実行して確認
- （B）シミュレータ（1機種×OS）
- （C）シミュレータ（アスペクト比が違う複数パターン）
- （D）実機（テスト対象端末の中から最も性能が低いもの×OS）
- （E）実機（テスト対象端末分）

## テスト項目：画面レイアウト

| NO | 詳細 | 観点 | 補足 | テスト方法 |
| --- | --- | --- | --- | --- |
| 1 | デザイン | 画面のレイアウトがデザイン通りであること（エラー発生時のレイアウトを含む） | - | A(Snapshot)、C |
| 2 | デザイン | スクロールが正しい領域に表示されていること（例えば、ページの一部の領域のみをスクロールするべきなのに、ページ全体がスクロールされていないかなど） | - | B |
| 3 | デザイン | OSのフォントサイズを大きくした場合などに文字が見切れていないこと（見切れる場合は、改行するか、末尾を「...」に表示にするなどの対策がされていること） | - | C |
| 4 | デザイン | 表示アニメーションが適切に表現されていること（画面遷移時のアニメーション、ダイアログやモーダル表示のアニメーション、一覧からアイテムを削除する場合のアニメーションなど） | - | C |
| 5 | 活性/非活性 | 活性となる状態の場合、該当UI部品が活性となっていること | - | B |
| 6 | 活性/非活性 | 非活性となる状態の場合、該当UI部品が非活性となっていること | - | B |
| 7 | 表示/非表示 | 表示の状態の場合、該当項目、文言が表示されること | - | B |
| 8 | 表示/非表示 | 非表示の状態の場合、該当項目、文言が表示されないこと | - | B |
| 9 | エラー | クライアントバリデーションエラーが発生した場合、対象のエラーが対象の項目に表示されること（複数件のエラーが発生した場合は、全てのエラーがその対象項目に表示されること） | - | B |
| 10 | エラー | サーバサイドバリデーションエラーが発生した場合、対象のエラーが対象の項目に表示されること（複数件のエラーが発生した場合は、全てのエラーがその対象項目に表示されること） | - | B |

## テスト項目：UI部品

| NO | 詳細 | 観点 | 補足 | テスト方法 |
| --- | --- | --- | --- | --- |
| 1 | データ | UI部品で表示するデフォルト値が設計通りであること | - | A |
| 2 | データ | UI部品で表示するデータの取得元が設計通りであること | - | A |
| 3 | データ | UI部品で表示するデータの編集仕様が設計通りであること | - | A |
| 4 | データ | UI部品で表示するデータのソート順が設計通りであること | - | A |

## テスト項目：一覧

| NO | 詳細 | 観点 | 補足 | テスト方法 |
| --- | --- | --- | --- | --- |
| 1 | - | 一覧の表示可能件数が上限以内であること | - | A |
| 2 | - | 一覧の表示件数が上限を超えた場合、以下の対応が実施されていること（ページング、スクロールで追加読込み（無限スクロール） | - | A |
| 3 | - | 一覧に表示されるデータのソート順が設計通りであること | - | A |

## テスト項目：画面表示データ

| NO | 詳細 | 観点 | 補足 | テスト方法 |
| --- | --- | --- | --- | --- |
| 1 | - | 画面に表示されるデータの取得元が設計通りであること | - | A(Mock)、B |
| 2 | - | 画面に表示されるデータが最大長の場合、エラーとならないこと | 最大件数と組み合わせて検証することを推奨する | A |
| 3 | - | 画面に表示されるデータが最大長の場合、レイアウト崩れがないこと | - | C |
| 4 | - | 画面に表示されるデータが最小長の場合、エラーとならないこと | - | A |
| 5 | - | 画面に表示されるデータが取得できない場合、設計通りとなること | - | A |
| 6 | - | サロゲートペア文字を許容する設計となっている場合、サロゲートペア文字を含む値を表示した結果がエラーとならないこと | 絵文字などを表示する | B |
| 7 | - | 画面に表示されるデータの件数が最大件数の場合、エラーとならないこと | - | A |
| 8 | - | 画面に表示されるデータの件数が最大件数の場合、レイアウトがデザイン通りであること | - | C |
| 9 | - | 画面に表示されるデータの件数が最大件数+1の場合、設計通りであること | - | A |
| 10 | - | 画面に表示されるデータの件数が最小件数の場合、エラーとならないこと | - | A |
| 11 | - | 画面に表示されるデータの件数が最小件数の場合、レイアウトがデザイン通りであること | - | C |
| 12 | - | 画面に表示されるデータの件数が最小件数マイナス1の場合、設計通りであること | - | A |
| 13 | - | 画面に表示されるデータの件数が0件の場合、レイアウトがデザイン通りであること | - | C |
| 14 | - | 画面に表示されるデータの件数が0件の場合、設計通りであること | - | A |
| 15 | - | 画面に表示されるデータの編集仕様が設計通りであること | 例：日付、姓名、金額 | A |
| 16 | - | 画面に表示されるデータのソート順が設計通りであること | - | A |
| 17 | 編集仕様 | 取得した値を変換する設計となっている場合、表示された値が設計通りであること | コードの変換、トリム処理、文字列の一部を取得する処理などを検証する | A |
| 18 | 編集仕様 | 取得した値を編集する処理が分岐する設計となっている場合、表示された値が設計通りであること | 分岐条件を網羅する | A |
| 19 | 編集仕様 | ループ処理によって、取得した値の編集を繰り返す設計となっている場合、表示された値が設計通りであること | 繰り返しの回数に応じた観点も含める、1つの値に対して繰り返し処理する場合も検証する、取得データが複数件あり、その件数分の処理を繰り返す場合も検証する | A |
| 20 | 編集仕様 | 取得した値を使用して計算する設計となっている場合、表示された値が設計通りであること | 桁あふれなども観点に含める | A |
| 21 | 編集仕様 | 取得した値を連結する設計となっている場合、表示された値が設計通りであること | 例：メールアドレス、電話番号 | A |
| 22 | 編集仕様 | 取得した値を暗号化する設計となっている場合、表示された値が暗号化されること | - | A |
| 23 | 編集仕様 | 取得した値をマスキングする設計となっている場合、表示された値がマスキングされること | - | A |

## テスト項目：エラー

| NO | 詳細 | 観点 | 補足 | テスト方法 |
| --- | --- | --- | --- | --- |
| 1 | - | エラーが発生した場合、リトライ可能なUIが提供されていること（一覧に表示する情報を取得するのに失敗した場合、WebViewに表示するページの取得に失敗した場合など） | - | B |
| 2 | - | エラーが発生した場合、ログの出力が設計通りになっていること | - | A |
