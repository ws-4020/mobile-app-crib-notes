---
title: テスト方針
---

サンプルアプリケーション開発における全体のテスト方針になります。
テスト方針として、`テストの種類`、`品質方針`、`優先度`を以下のとおりまとめます。

## テストの種類

このサンプルアプリケーションでは、以下のテストを実施します。

### 単体テストの種類

- テストコードによる自動テスト
- iOSとAndroidのシミュレータ／エミュレータまたは実機を用いた手動テスト

### 結合テストの種類

- シミュレータまたは実機を用いた手動テスト

このアプリケーションはサンプルという位置づけであり、納品先ユーザが存在していません。
そのため、結合テストまでの結果を、後述する品質方針に則って評価します。

## 品質方針

このサンプルアプリケーションでは、以下の観点で品質を評価し、最終的な合否判定を実施します。

- 定量評価
- 定性評価
- 非機能要件の評価
- 後方互換性の評価
- 利用者からのフィードバックと対応状況の評価
- リリース後、不具合の発生状況

以下に各評価の詳細を記載します。

### 定量評価

ドキュメントとソースコードそれぞれのレビューや不具合実績等を以下のとおり集計します。

| NO | 定量値 | 区分 |
| --- | :--- | :--- |
| 1 | レビュー実施率 | ドキュメント、ソースコード |
| 2 | レビュー工数密度 | ドキュメント、ソースコード |
| 3 | レビュー指摘密度 | ドキュメント、ソースコード |
| 4 | テストケースレビュー実施率 | 単体テスト、結合テスト |
| 5 | 単体テストケース、結合テストケース密度 | 単体テスト、結合テスト |
| 6 | 単体テストケース、結合テストケース不具合の検出密度 | 単体テスト、結合テスト |

上記それぞれの集計値と、比較対象となる過去のプロジェクトでの定量値を比較して評価します。

### 定性評価

参画メンバーのスキルバランス妥当性、作成物のマージ時におけるレビュー体制の妥当性、開発プロセスの妥当性について評価します。

### 非機能要件の評価

作成物と非機能要件を照らし合わせ、要件に沿っているかについて評価します。

### 後方互換性の評価

後方互換性の必要有無、必要な場合は考え方が妥当かどうかについて評価します。

### 利用者からのフィードバックと対応状況の評価

レビューアからのフィードバックに対する評価とフィードバックへの対応内容について評価します。

### リリース後、不具合の発生状況

過去リリースから今回リリースまでに生じた不具合とその対応内容について評価します。

## テスト実施範囲

テストとして、単体テストと結合テストを実施します。それぞれのテスト範囲の概要は以下のとおりです。

### 単体テストの実施範囲

開発した画面単位または、機能単位での動作検証。

### 結合テストの実施範囲

今回開発した画面または、機能および、連携するサービスを含んで網羅的なテストを実施します。

## 優先度

今回のサンプルアプリケーションは、段階的に開発を進めていきます。
開発全体から、以下の観点で優先度をつけて、開発および、テストを進めていきます。

- サンプルアプリケーションとして、段階的に開発を進めるため、作成物を早期にイメージしやすい部分
- アーキテクチャ部分の部品開発と並行しながら、このサンプルアプリケーション開発を進めるため、部品開発の順序に合わせて取込みやすいもの
